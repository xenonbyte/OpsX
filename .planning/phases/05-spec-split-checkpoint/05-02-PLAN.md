  <action>Implement D-04 through D-07 in a new CommonJS module `lib/spec-validator.js`. Export exactly `collectSpecSplitEvidence(options = {})`, `parseSpecFile(filePath, text)`, and `reviewSpecSplitEvidence(evidence, options = {})`. Do not add a Markdown parser dependency; use built-in string scanning only. `parseSpecFile()` must ignore `### Requirement:` and `#### Scenario:` headings that appear inside fenced code blocks, but it must collect those fence bodies separately for hidden-requirement detection. Treat the normalized `### Requirement:` title as the requirement id for duplicate-id detection. `collectSpecSplitEvidence()` must parse proposal scope from `## What Changes`, `## Capabilities`, `### New Capabilities`, and `### Modified Capabilities`, then return a structured corpus with per-file `path`, `capability`, `requirements`, `scenarioCount`, `fencedBlocks`, and aggregate counts. `reviewSpecSplitEvidence()` must emit stable finding codes with existing-artifact patch targets only: `proposal-coverage-gap`, `scope-expansion-unapproved`, `duplicate-requirement-id`, `duplicate-behavior-likely`, `conflicting-requirements`, `spec-empty`, `scenario-missing`, and `hidden-requirement-in-fence`. Use these deterministic heuristics: duplicate ids are matching normalized requirement titles; duplicate behavior is a cross-file or cross-section pair with different titles but Jaccard token overlap >= 0.75 on requirement body tokens; conflicts are duplicate-id or duplicate-behavior pairs where one requirement contains `must not`, `shall not`, `never`, or `cannot` and the other contains a positive normative phrase over overlapping noun tokens; hidden requirements are fenced blocks that contain `### Requirement:`, `#### Scenario:`, or `SHALL`/`MUST`; empty specs are files with zero valid requirements outside fences; missing scenarios are requirements with zero `#### Scenario:` headings outside fences; proposal coverage gaps occur when a capability line from proposal scope has no token overlap with any parsed requirement; unapproved scope expansion occurs when a requirement introduces three or more non-stopword tokens with zero overlap to proposal scope tokens. Findings must target `proposal`, `specs`, or the concrete spec file paths, never a new `spec-review.md` artifact.</action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| markdown artifact text -> validator | malformed headings or fenced-code tricks can hide requirements or distort coverage checks |
| proposal scope -> parsed spec corpus | scope terms can be spoofed or expanded silently if parsing is too loose |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03 | Spoofing / Tampering | `lib/spec-validator.js` markdown parsing | mitigate | Parse valid requirements outside fences only and scan fenced blocks separately for hidden normative content. |
| T-05-04 | Tampering | duplicate/conflict detection | mitigate | Use deterministic normalized titles and token-overlap heuristics with explicit test fixtures and file-specific patch targets. |
| T-05-05 | Elevation of Scope | proposal/spec scope comparison | mitigate | Build proposal-scope tokens from constrained sections and emit explicit `proposal-coverage-gap` / `scope-expansion-unapproved` findings. |
</threat_model>
